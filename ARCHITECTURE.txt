SITE PLAN SUBSYSTEM (the part that keeps getting rewritten -- READ THIS FIRST)
================================================================================

CURRENT ARCHITECTURE: S79 polygon-edge rotation. NOT the S73 coordinate
unification (deleted by S78/S79 -- commits feb4eee, 1e40b4c). Any context
file claiming S73 is the live architecture is stale. Philosophy:
"good enough, never wrong" -- optimize the floor, dimensions are sacred,
position is flexible, never push the house to a lot edge.

DATA FLOW:
1. Realie parcel lookup returns lot vertices + address geocode.
2. Parcel callback (steps.js ~1000-1190):
   a. Street edge detected: ray from lot centroid through address point;
      lowest-avg-Y edge as fallback. Rear edge = reversed ray.
   b. Rotation computed FROM THE STREET EDGE ANGLE of the polygon itself
      (zero external dependency): rot = -atan2(edgeDy, edgeDx), flipped 180
      if street midpoint would land above centroid. Skipped when < 3 deg.
   c. Vertices rotated around centroid, shifted so min corner = (0,0).
      Transform params kept in closure (_s79Cos/_s79Sin/_s79Cx/_s79Cy/
      _s79Mx/_s79My/_s79Rotated) for reuse by the footprint callback.
   d. lotVertices set LAST (engine regenerates polygon when null -- race).
3. Building footprint callback (Overpass, async, ~3-15s, VALIDATION role):
   a. If Overpass road bearing disagrees with ray-cast street edge:
      re-rotate from original vertices, update closure vars + state.
      If Overpass fails: keep ray-cast result (medium confidence).
      Auto-retry once after 2s; warning banner + manual retry if both fail.
   b. Primary residence = closest building >= 400 sqft within 250 ft.
   c. House positioning ladder: building centroid (if inside polygon)
      -> address point -> centered fallback. Offset measured from
      tightEdges().left (tightest span across house top/mid/bottom Ys).
   d. Auto-detected values stashed: _autoHouseOffset/_autoHouseDist/
      _autoHouseWidth/_autoHouseDepth (drives the green banner + reset UI
      + AI resetHousePosition action + engine clamp bypass).
4. houseAngle is ALWAYS 0 under S79. No building tilt, no width/depth swap.

RENDERER (sitePlanView.js): house X = leftEdgeAtY(houseMidY) + houseOffsetSide.
PDF (draw_site_plan.py): same values, same leftEdgeAtY approach.

RESOLVED S85: houseOffsetSide is RENDERER-SPACE. positionHouse() converts
its tightEdges-based placement to the leftEdgeAtY(midY) reference at stash
time (see lotGeometry.js OFFSET CONTRACT header). intended==rendered by
construction; H5 in the geometry suite is a hard check. steps.js and
sitePlanView.js call sites now delegate to window.lotGeometry (index.html
loads lotGeometry.js, cache buster s85a).

PURE GEOMETRY MODULE (S84): backend/static/js/lotGeometry.js is the faithful
extraction of all of the above math. Tests: tests/geometry/ (2700+ invariant
checks, all shapes x all rotations, runs in CI). RULE: any change to site

steps.js/sitePlanView.js still contain the original inline copies; swapping

SHEET SIZE + COMPLEXITY BRANCH (S86 -- READ BEFORE TOUCHING SHEET LAYOUT)
================================================================================
SHEET SIZE is a parameter, not a literal. backend/drawing/sheet.py is the
single source: SHEET_SIZES registry (letter 14x8.5 / tabloid 17x11 / arch_c
24x18 / arch_d 36x24), sheet_size() getter, and render_scale() -- a context
manager that scales every point-valued style (font.size, lines/patch/hatch
linewidth, markersize, and explicit fontsize kwargs on Axes.text/annotate/
Figure.text) by font_scale() = sheet_width / 14. This keeps normalized type
constant as the sheet grows (sheets are drawn in DATA feet with fonts in
absolute POINTS, so a bigger sheet would otherwise shrink text relatively).
render_scale() is a guaranteed NO-OP at scale 1.0 (14x8.5). DEFAULT ships
arch_d 36x24. Override per-render with SBP_SHEET env var or set_active().
main.py wraps both PdfPages blocks in `with render_scale():` and creates every
figure at figsize=sheet_size().

COMPLEXITY BRANCH (main.py generate_blueprint_pdf): _complex = deck has >=1
zone. The permit sheet sequence is data-driven:
  simple:  A-1 DECK PLAN & FRAMING (combined), A-2 elev, A-3 notes, A-4
           details, A-5 site, A-6 checklist.
  complex: A-1 DECK PLAN, A-2 DECK FRAMING (each a full-width single panel),
           A-3 elev, A-4 notes, A-5 details, A-6 site, A-7 checklist.
draw_plan_and_framing(fig, params, calc, spec, panels=...) drives this:
panels=("plan","framing") is the historical side-by-side (fig.subplots(1,2));
("plan",) or ("framing",) draws one full-width panel (fig.subplots(1,1)). The
cover drawing-index (draw_cover_sheet sheet_index=) and the draw_notes zone
prose reference the numbering, so both branches stay internally consistent.
INVARIANT: the simple (no-zone) path is byte-for-byte identical to pre-S86
(verified). Any change here must preserve that -- test by rendering a no-zone
config and diffing pixel hashes against a pre-change render.

FRAMING CALLOUT PLACEMENT (draw_plan.py, is_framing branch): member callouts
live OFF the geometry. Beam group (posts/piers, beam ply, hardware, blocking)
are right-anchored just off the drawing's left edge on short leaders via
_margin_callout(). Zone-wing specs print just below each wing. DECK LOADS is a
titled table (green header, label/value columns, ruled total) anchored lower-
left above the scale bar. draw_zone_framing(..., margin_cols=) toggles the
complex-sheet wing labeling. KNOWN: this per-wing scheme is unverified past ~2
zones; 4+ zones may need a consolidated framing schedule (S87 item 3).
