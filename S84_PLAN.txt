================================================================================

2.1 GEOMETRY TEST HARNESS (highest-leverage engineering task, S84)
The site plan subsystem was rebuilt in S70, S72, S73, S78, S79 -- five
rewrites, each "fixing one address while breaking another" (S79's own words).
Root cause: pure math verified by clicking addresses in a browser.

Plan:
a. Extract the S79 geometry into a standalone pure module
   (backend/static/js/lotGeometry.js + Python mirror or shared logic):
   - streetEdge detection (ray-cast)
   - polygon rotation + origin normalization
   - leftEdgeAtY / tightEdges scans
   - point-in-polygon, house placement + nudge logic
   No React, no window.*, no closures over component state. Inputs in,
   outputs out.
b. Parameterized tests (Node script or ported to Python/pytest):
   - Synthetic lots: rectangle, trapezoid, flag lot, concave, tapered,
     at rotations swept 0-359 degrees in 5-degree steps
   - Invariants asserted for every case:
     * street edge lands at bottom (min Y) after rotation
     * house rect fully inside polygon (or flagged, never silently clipped)
     * houseOffsetSide >= 0, houseDistFromStreet >= 0
     * round-trip: rotate then un-rotate reproduces original vertices
   - Regression fixtures: the actual Realie payloads for the known problem
     addresses (10 Chichester, 368 Croaton, etc). Save the JSON responses as
     fixtures so past bugs become permanent tests.
c. This is the Fable 5 experiment Will asked about. Realistic expectation:
   the model is genuinely good at this exact task (extraction, refactoring,
   symbolic verification of rotation math, exhaustive parameterized tests).
   What it cannot do: validate against ground truth (real surveys) or
   guarantee a plans examiner accepts the output. The S72/S73/S78/S79 churn
   was mostly a PROCESS failure (no tests, verify-by-clicking, context
   drift), so harness + model together is the fix, not model alone.
