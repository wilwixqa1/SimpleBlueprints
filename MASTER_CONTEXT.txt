================================================================================
1. WHAT SIMPLEBLUEPRINTS IS
================================================================================

SimpleBlueprints (simpleblueprints.xyz) is a permit-ready deck blueprint
generator. Users enter their address, the system auto-detects lot shape and
house position from parcel/satellite data, they configure a deck, and the
system produces a multi-page IRC 2021-compliant permit PDF package: cover
(3D render), plan & framing, elevations, general notes, structural details,
site plan, deck attachment checklist, materials list.

Owner-developer: Will Wendt (sole developer), Ronkonkoma NY (Suffolk County).
Business partner Billy handles community validation/marketing (recovering
from illness, coming back online). Live in beta. NO paying customers yet.
Stripe wired but not active. Price currently $29 (main.py BLUEPRINT_PRICE=2900).

TARGET CUSTOMER (decided S84): CONTRACTORS, not DIY homeowners. Implications:
pricing likely $49-99/plan or subscription; PDF correctness is the whole
product; contractors have surveys in hand and examiner relationships.

THE BUSINESS MILESTONE THAT MATTERS: one generated package approved by a
real building department. Nothing else counts as validation. TARGET
JURISDICTION (corrected S86): COLORADO SPRINGS / Pikes Peak Regional Building
Dept (PPRBD) -- partner Billy operates there, and the professional reference
sets (All Things Architecture) are CO Springs / PPRBD permit sets. The old
Brookhaven/Islip (Suffolk County NY) framing from S84 is SUPERSEDED. The code
already has is_colorado_springs() + append_cos_attachment() (the official
PPRBD deck-attachment form, letter size, merged as-is onto the set).

FEATURE FREEZE (agreed S84): no new capabilities (no sheds, no pergolas, no
new zone types or stair templates) until that milestone is hit. calc_shed.py
and calc_pergola.py exist in the repo and stay dormant.

6.4 VISION OUTAGE MODE + STRATEGY (S85 first observed; corrected S87)
- THE FAILURE: Claude's image VIEWING dies mid-session while image
  GENERATION keeps working (PDFs render, pdftoppm makes PNGs fine; the
  view tool just returns EMPTY). Does NOT recover within a session; a
  FRESH session restores it.
- CROSS-SESSION PATTERN (this is the important correction): it is NOT a
  fixed image count and NOT per-image depletion. S85 = worked for 40+
  views then died. S86 = ~10 then died. S87 = died after just ONE view.
  What predicts death is total SESSION CONTEXT SIZE, not images viewed:
  S87's context was already heavy (clone + 4 context files + suites +
  multiple renders) BEFORE the single view. Best current model: once
  cumulative context crosses some threshold, tool-result image payloads
  stop coming back. THIS IS THEORY, NOT CONFIRMED FACT -- Claude cannot
  introspect the real mechanism. Do NOT write it as certain, and do not
  let a wrong theory ossify; it just happens to fit 3 sessions.
- THE STRATEGIC CONSEQUENCE (drives how S88+ should be run): if the driver
  is context size, the fix is TWO-PRONGED and neither prong needs the
  theory to be exactly right:
  1. VISION-FIRST SESSION ORDERING (CONFIRMED S88 -- it worked).
     Every session BEFORE S88 front-loaded HOUSEKEEPING (clone, read 4 files,
     run suites, render) and only THEN did visual review -- by which point
     context is already heavy and vision is often dead. S88 FLIPPED IT: the
     very first actions were clone, install, render ONLY the sheets needing
     eyes, and VIEW them -- before reading MASTER_CONTEXT, before suites.
     RESULT: 17/17 views with ZERO drops (vs S87's 1-then-dead, S86's ~10,
     S85's 40+). This is the strongest evidence yet that the driver is
     cumulative context size, not image count -- and the cheap, falsifiable
     fix held up. STANDING PRACTICE NOW: any session with visual work renders
     + views the needed sheets FIRST. Still THEORY on the mechanism (don't
     write the cause as certain), but the ORDERING FIX is proven useful.
  2. NEED VISION LESS by building oracles (the durable fix -- see the S87
     KEY INSIGHT in 10D). Convert every recurring EYE-judgment that is
     really a geometric fact into a detector. Done so far: legibility
     (text-over-text, S85), text-over-linework (S87 p6), post-on-grade
     (S87 p10). Next to build: text-outside-panel, text-clipped-at-edge,
     callout-into-neighbor-view. Each one permanently removes a reason to
     need eyes.
  3. SMALLER IMAGES (minor, independent lever): review PNGs are 5400x3600.
     Rendering at 96 DPI instead of 150, or CROPPING to the region under
     review instead of viewing a whole 36x24 sheet, is cheaper per image =
     more views before the threshold. Low cost, worth doing.

  BEFORE heavy context accumulates (see ordering above). If dead, know it
  before planning visual work.
- WORKAROUNDS when vision is down, in order of strength:
  (a) Will screenshots into chat -- CHAT-UPLOADED IMAGES STILL REACH CLAUDE

7. THE ORACLE STACK (testing design, agreed S84 -- S85 builds the missing parts)
================================================================================

Layered verification, cheap-to-expensive; goal is "representative of input
AND actually good," not just compliant:

  L1 CRASH ORACLE: random valid configs -> pipeline must not throw.
     Scales to thousands of configs. (Script needed: S85.)
  L2 COMPLIANCE ORACLE: permit_checker.run_checks -- 143 IRC checks.
     Already exists; it is the built-in "judge."
  L3 CONSISTENCY ORACLE (S85): programmatic input-vs-PDF checks:
     - spec values from permit_spec must appear on rendered sheets
     - count of drawn stairs == len(deckStairs); labeled elevation side
       matches projected geometry; dims on sheets == input params
     - TEXT COLLISION DETECTOR: matplotlib knows every text bbox; a
       post-render hook flags any overlapping text bboxes. Catches
       "words on top of each other" deterministically at scale.
  L4 VISUAL ORACLE: model views rasterized pages against a rubric
     (correct views, nothing clipped, proportions sane, drawing looks like
     the configured deck). Does not scale to thousands; use a stratified
     sample (~20-40 configs covering attachment x zones x stairs x lot
     shapes) per run. Failing seeds from any layer become permanent
     regression fixtures.
  GOLDEN FILES (S85): fixed-seed renders compared as pixels; human blesses
     each golden once.

8B. LOOP ENGINEERING DOCTRINE (S85 -- how we build with loops, and when NOT to)
================================================================================

THE PATTERN (proven S85): define a loop as GOAL + MACHINE JUDGE + BUDGET +
HUMAN GATE before writing code.
  GOAL: a precise, machine-checkable end state ("geometry suite 0 failures
    AND 0 divergence warnings", "<=2 genuine collisions per set").
  MACHINE JUDGE: the deterministic thing that says pass/fail without taste
    (test suite, legibility detector, instrumented coordinates, DOM
    extraction, permit checker). If no such judge exists, BUILD THE JUDGE
    FIRST or don't call it a loop.
  BUDGET: iteration cap (S85 used 10; divergence fix converged in 2).
    Hitting the cap = stop and report, never grind silently.
  HUMAN GATE: Will's eyes on rendered output before anything examiner-
    facing ships. Machine green is necessary, not sufficient.

WHEN LOOPS ARE APPROPRIATE:
  - Mechanical objectives with a crisp oracle: make tests green, drive a
    measured defect count to a threshold, fuzz for crashes, converge two
    implementations (divergence-style bugs), regression campaigns.
  - Battle-testing config matrices where each run auto-judges through the
    oracle stack (crash -> permit checks -> legibility -> visual sample).

WHEN LOOPS ARE NOT APPROPRIATE (STANDING RULE, requested by Will S85):
  If Will asks for a loop at an inappropriate time, Claude MUST flag it
  rather than comply silently. Inappropriate means:
  - Taste / design / examiner-facing judgment calls (grade-line style,
    cover aesthetics, annotation layout STYLE). Correct form: prototype
    2-4 options, render, present for Will's pick. S85 grade styles worked
    exactly this way.
  - No deterministic judge exists and building one isn't feasible --
    looping against vibes just oscillates.
  - One-shot decisions (sheet size, pricing, targeting) -- these are
    decisions, not convergence problems.
  - Vision down + the only judge is visual: do NOT blind-loop on
    appearance. Queue it or get Will's screenshots per iteration.

S85 LESSONS LEARNED (encode these):
  1. LOOPS CONVERGE FAST WHEN THE JUDGE IS RIGHT: divergence fix hit
     goal-state in 2 iterations because the harness measured exactly the
     defect.
  2. BLIND ITERATION OSCILLATES: the dimension-clearance fix went
     4 -> 6 -> 9 collisions across three shotgun attempts until stopping,
     pulling the COMPLETE defect list, and fixing once (landing-pad extent
     was the missing fact). Rule: after 2 iterations moving the wrong way,
     STOP and gather the full picture.
  3. JUDGE QUALITY IS PART OF THE LOOP: the detector needed calibration
     (CAD-duplicate filter, 30% threshold) before its numbers meant what
     we thought. Recalibrating the judge is legitimate loop work;
     redefining the goal to dodge failures is not.
  4. ORACLES CATCH WHAT EYES MISS AND VICE VERSA: the detector found the
     universal A-6 collision eyes had shrugged off; Will's eyes found the
     floating piers, panel overflow, and clipped callouts no oracle saw.
     Both layers are mandatory.
  5. FIXING GEOMETRY EXPOSES LAYOUT DEBT: correct stair projection RAISED
     collision counts on crowded configs. A metric temporarily worsening
     because correctness improved is progress -- say so explicitly with
     the causal story, never hide it.
  6. EVERY LOOP FAILURE BECOMES A PERMANENT FIXTURE immediately (harness
     param set or test), then fix the worst 1-2 and backlog the rest with
     reproducers. No rabbit-holing the whole list in one session.
