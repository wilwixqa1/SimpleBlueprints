================================================================================
1. WHAT SIMPLEBLUEPRINTS IS
================================================================================

SimpleBlueprints (simpleblueprints.xyz) is a permit-ready deck blueprint
generator. Users enter their address, the system auto-detects lot shape and
house position from parcel/satellite data, they configure a deck, and the
system produces a multi-page IRC 2021-compliant permit PDF package: cover
(3D render), plan & framing, elevations, general notes, structural details,
site plan, deck attachment checklist, materials list.

Owner-developer: Will Wendt (sole developer), Ronkonkoma NY (Suffolk County).
Business partner Billy handles community validation/marketing (recovering
from illness, coming back online). Live in beta. NO paying customers yet.
Stripe wired but not active. Price currently $29 (main.py BLUEPRINT_PRICE=2900).

TARGET CUSTOMER (decided S84): CONTRACTORS, not DIY homeowners. Implications:
pricing likely $49-99/plan or subscription; PDF correctness is the whole
product; contractors have surveys in hand and examiner relationships.

THE BUSINESS MILESTONE THAT MATTERS: one generated package approved by a
real building department. Nothing else counts as validation. Plan: informal
plans-examiner review at a Suffolk County town (Brookhaven or Islip) first.

FEATURE FREEZE (agreed S84): no new capabilities (no sheds, no pergolas, no
new zone types or stair templates) until that milestone is hit. calc_shed.py
and calc_pergola.py exist in the repo and stay dormant.

2. STACK AND ARCHITECTURE
================================================================================

FRONTEND: Single-page React JSX transpiled in-browser via Babel standalone,
NO build step. 13 JS files in backend/static/js/. State = one "p" object;
u(key, value) mutates through the app.js reducer (reducer has side effects;
always check it when stored values change unexpectedly). Helpers on window.*.
Three.js for 3D (deck3d.js). Per-file cache busters in index.html (?v=s81e
etc.) -- DRIFTED, consolidation to single global string is a TODO.

BACKEND: Python FastAPI on Railway, autodeploys from main on push. NO staging.
PDF generation via matplotlib + PdfPages (backend/drawing/, ~15K lines).
PostgreSQL (Railway plugin). Google OAuth. Stripe (inactive). Realie API
(parcel boundaries). Overpass/OSM (building footprints, flaky ~30%).
Anthropic API (AI helper chat, survey extraction).

REPO: github.com/wilwixqa1/SimpleBlueprints (public).
CI (added S84): .github/workflows/tests.yml runs three suites on every push.

ENV VARS (Railway): REALIE_API_KEY, STRIPE_* (3), GOOGLE_CLIENT_ID/SECRET,
ANTHROPIC_API_KEY, DATABASE_URL, SESSION_SECRET, ADMIN_PASSWORD (now REQUIRED
-- S84 made admin fail-closed; unset = admin endpoints return 503).

4. THE DATA MODEL (the "p" object) -- unchanged since prior master file
================================================================================

DECK: width, depth, height, attachment ('ledger'|'freestanding'), joistDir,
joistSpacing (12|16|24), beamType ('dropped'|'flush', zone 0), deckingType,
frostZone, snowLoad, slopePercent/Direction.

ZONES: zones:[{id, type('add'|'cutout'), w, d, h(null=inherit), attachTo,
attachEdge, attachOffset, interiorY, corners, joistDir, beamType, label}],
activeZone. Zone 0 = main deck (virtual).

LOT+HOUSE: lotVertices (drawing space), lotEdges (street/rear/side metadata),
lotWidth/Depth, houseWidth/Depth, houseOffsetSide, houseDistFromStreet,
houseAngle (ALWAYS 0 under S79), northAngle, streetName, setbackFront/Side/
Rear, _autoHouseOffset/_autoHouseDist/_autoHouseWidth/_autoHouseDepth
(stashed auto-detected values: green banner + reset UI + AI reset action +
engine clamp bypass), _lotRotation (0 after rotation).

STAIRS: deckStairs:[{id, zoneId, location, width, template('straight'|
'l-left'|'l-right'|'platform'|'u-turn'|'wrap'), offset, anchorX/Y, angle,
numStringers, _landsOnZoneId (null=grade, number=lands on zone)}].

SITE: siteElements:[{type, x, y, w, d, label}].

6. CAPABILITY PLAYBOOK (S84 DISCOVERIES -- DO NOT LOSE THESE)
================================================================================

Claude chat sessions have a Linux sandbox with bash, Python, Node, network,
and VISION. Three capabilities every session should know:

6.1 GENERATE + VISUALLY REVIEW PDFs LOCALLY (no website, no user screenshots)
  Setup (sandbox resets each session, ~2 min):
    pip install matplotlib numpy httpx python-multipart itsdangerous \
        anthropic fastapi stripe psycopg2-binary --break-system-packages
  Generate:
    cd repo/backend && python3 -c "
    import sys; sys.path.insert(0,'.')
    from app.main import generate_blueprint_pdf
    out = generate_blueprint_pdf({...params...})   # returns 4 values, [0]=permit id
    "
    PDFs land in /tmp/blueprints/<uuid>.pdf (permit set is the larger file).
  Rasterize + LOOK:
    pdftoppm -png -r 150 <file>.pdf /tmp/page     # 150 DPI for text judgment
    then view the PNGs (actual vision, not text extraction).
  RULE (standing): PDF review happens by rasterizing and viewing. NEVER by
  text extraction. Historical failures ("Claude said the PDF was fine")
  came from text-extraction review. This is why the old rule "never confirm
  a visual fix from text extraction alone" exists.
  LIMITS: model vision is reliable for layout-level judgment (missing/
  overlapping/clipped/mispositioned elements), weaker for fine aesthetics.
  Will's eyes remain the final gate for examiner-facing output.

6.2 DRIVE THE LIVE SITE IN A HEADLESS BROWSER (UI/UX review without Will)
  Setup (~2-3 min):

7. THE ORACLE STACK (testing design, agreed S84 -- S85 builds the missing parts)
================================================================================

Layered verification, cheap-to-expensive; goal is "representative of input
AND actually good," not just compliant:

  L1 CRASH ORACLE: random valid configs -> pipeline must not throw.
     Scales to thousands of configs. (Script needed: S85.)
  L2 COMPLIANCE ORACLE: permit_checker.run_checks -- 143 IRC checks.
     Already exists; it is the built-in "judge."
  L3 CONSISTENCY ORACLE (S85): programmatic input-vs-PDF checks:
     - spec values from permit_spec must appear on rendered sheets
     - count of drawn stairs == len(deckStairs); labeled elevation side
       matches projected geometry; dims on sheets == input params
     - TEXT COLLISION DETECTOR: matplotlib knows every text bbox; a
       post-render hook flags any overlapping text bboxes. Catches
       "words on top of each other" deterministically at scale.
  L4 VISUAL ORACLE: model views rasterized pages against a rubric
     (correct views, nothing clipped, proportions sane, drawing looks like
     the configured deck). Does not scale to thousands; use a stratified
     sample (~20-40 configs covering attachment x zones x stairs x lot
     shapes) per run. Failing seeds from any layer become permanent
     regression fixtures.
  GOLDEN FILES (S85): fixed-seed renders compared as pixels; human blesses
     each golden once.

8. GUIDED WIZARD, ENGINE, ZONES, PDF PIPELINE (stable subsystems)
================================================================================

WIZARD: 4 steps in steps.js (0 site plan, 1 deck config, 2 materials,
3 review/download). Guide via _guidePhaseMap (phases show/hide sections,
action buttons). Step 0 flow: address_lookup -> address_verifying ->
footprint_loading -> verify_extracted -> site_elements_check -> ...
AI chat can mutate p via actions (setParam, addZone, cutoutAdd, addStair,
resetHousePosition, ...); backend builds context prompt in main.py.

ENGINE: engine.js computes joists (R507.5), beams, posts (4x4/6x6),
footings (frost zone), post counts per IRC 2021. calc_engine.py mirrors it
for PDF (dual-implementation tax; S82e post-count mismatch was this class).
calcAllZones: flush zones = joists only (rim board as beam); flush
auto-converts to dropped when zone.h != mainH (getEffectiveBeamType).

ZONES: zone 0 virtual main deck. Key window.* helpers in zoneUtils.js:
getAllZones, getZoneRect, getCompositeOutline, getExposedEdges,
getSharedEdges (single source of truth for internal boundaries),
classifyHeightDelta/suggestRiserPlan (own ALL IRC thresholds, never
hardcode). Transitional stairs: _landsOnZoneId; rise from height delta;
PDF labels "LANDS ON [zone]".

PDF PIPELINE: POST /api/calculate -> calculate_structure ->
build_permit_spec -> run_checks -> draw_* per sheet -> PdfPages.
Sheets: A-0 cover, A-1 plan/framing, A-2 elevations, A-3 notes,
A-4 details, A-5 site plan, A-6 attachment checklist, A-7 materials.
Steel: CCRR-0313 tables; steel posts never buried; Welborn labeling.

API ENDPOINTS: unchanged from prior master file -- auth (Google OAuth),
/api/calculate, /api/extract-survey, /api/rank-shapes, /api/ai-helper,
/api/check-permit, /api/parcel-lookup, /api/building-footprint, projects
CRUD (user-scoped), Stripe checkout/check-payment/download, tracking,
admin (now fail-closed + rate limited).
