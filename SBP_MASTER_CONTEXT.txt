================================================================================
1. WHAT SIMPLEBLUEPRINTS IS
================================================================================

SimpleBlueprints (simpleblueprints.xyz) is a permit-ready deck blueprint generator. Users enter their address, the system auto-detects their lot shape and house position from satellite/parcel data, they configure a deck, and the system produces a multi-page IRC 2021-compliant permit submission PDF package: cover sheet (with 3D render), site plan, framing plan, deck plan, elevations, structural details, general notes, materials list, and compliance checklist.

Target users: DIY builders, general contractors, and architects who need a permit package they can hand to a building official without hiring a structural engineer for standard deck shapes.

Owner-developer: Will Wendt (sole developer). Business partner handles community validation and marketing. Will is based in Ronkonkoma, NY.

The product is live at simpleblueprints.xyz (beta). No paying customers yet. Revenue model: per-plan purchase via Stripe.

4. THE DATA MODEL (the "p" object)
================================================================================

p is the single React state object. Key fields:

DECK GEOMETRY:
  width, depth, height          Main deck dimensions (ft)
  attachment                    'ledger' | 'freestanding'
  joistDir                      'perpendicular' | 'parallel'
  joistSpacing                  12 | 16 | 24 (inches o.c.)
  beamType                      'dropped' | 'flush' (zone 0 only)
  deckingType                   'composite' | 'pt_lumber'
  frostZone, snowLoad           Climate parameters
  slopePercent, slopeDirection  Deck slope

ZONES (S60+, multi-level decks):
  zones: [{
    id, type,                   'add' | 'cutout', id never reused
    w, d, h,                    h: ft override or null (null = inherit main)
    attachTo, attachEdge,       Parent zone id, edge
    attachOffset, interiorY,
    corners,                    Chamfer config per corner
    joistDir, beamType,         Per-zone, independent
    label                       User-editable (S80b)
  }]
  activeZone                    Currently selected zone (0 = main)

LOT + HOUSE (S70+):
  lotVertices                   Array of [x,y] pairs (drawing space after S73)
  lotEdges                      Edge metadata (street/rear/side labels, lengths)
  lotWidth, lotDepth            Bounding box of lot
  houseWidth, houseDepth        Building footprint dimensions
  houseOffsetSide               Distance from left polygon edge (drawing space after S73)
  houseDistFromStreet           Y position in drawing space (after S73)
  houseAngle                    Building tilt angle (SVG convention, negated)
  northAngle                    For north arrow on site plan
  streetName                    From Overpass road data
  setbackFront/Side/Rear        From zoning code

  _autoHouseOffset              Stashed auto-detected offset (S73, for reset)
  _autoHouseDist                Stashed auto-detected distance (S73, for reset)
  _autoHouseWidth               Stashed auto-detected house width (S73)
  _autoHouseDepth               Stashed auto-detected house depth (S73)
  _lotRotation                  Set to 0 after rotation (prevents double-rotate)

STAIRS:
  deckStairs                    Array of stair objects [{
    id, zoneId,                 Which zone this stair belongs to
    location,                   'front' | 'left' | 'right' | 'rear'
    width, template,            'straight' | 'l-left' | 'l-right' | 'platform' | 'u-turn' | 'wrap'
    offset, anchorX, anchorY, angle,
    numStringers,
    _landsOnZoneId              null = grade, number = lands on that zone (transitional, S81d)
  }]

SITE ELEMENTS:
  siteElements                  Array of {type, x, y, w, d, label} for garage, pool, etc.

5. HOW THE SITE PLAN WORKS (S70-S73, CRITICAL)
================================================================================

!! S73 HIGH-RISK CHANGE WARNING !!
S73 made a fundamental change to how house positioning works for rotated lots.
Only verified on ONE property (10 Chichester Rd, lotRotation=166).
If you encounter bugs with house position, lot orientation, setback gaps, or PDF
output, S73 coordinate unification is the most likely cause.

DATA FLOW:
1. User enters address, Realie API returns lot boundary vertices + address point
2. Street edge detected via centroid-to-address-point ray
3. Building footprint API (Overpass/OSM) fires async:
   a. Road correction: identifies correct street edge via nearest road bearing
   b. Computes drawRotation to put street at bottom
   c. If drawRotation != 0:
      - Rotates lot vertices around centroid, normalizes to origin
      - Computes house position in un-rotated space, then rotates the output
      - S73: Recomputes houseOffsetSide and houseDistFromStreet in drawing space
      - All downstream consumers see values consistent with rotated polygon
      - No consumer knows rotation occurred
   d. Stashes auto-detected values (_autoHouseOffset, _autoHouseDist, etc.)

CRITICAL RULES:
- Set lotVertices LAST in u() call sequences (engine regenerates polygon if null)
- Do not re-derive house position from rotated data
- Set _autoHouseDist BEFORE houseDistFromStreet (engine clamp bypass)
- houseAngle is stored in SVG convention (negated for Y-flip)
- The engine (app.js) clamps houseDistFromStreet >= setbackFront UNLESS _autoHouseDist exists

RENDERER (sitePlanView.js):
- Single code path: leftEdgeAtY(houseMidY) + houseOffsetSide
- No _houseX/_houseY variables (eliminated in S73)
- Gap DimLines computed geometrically from polygon edges

PDF BACKEND (draw_site_plan.py):
- Receives same houseOffsetSide/houseDistFromStreet/lotVertices
- Runs same leftEdgeAtY computation
- houseAngle used directly (matplotlib Y-up happens to produce correct visual)

6. HOW THE GUIDED WIZARD WORKS
================================================================================

The wizard has 4 steps (step 0-3):
  Step 0: Site Plan (address lookup, lot/house config, site elements)
  Step 1: Deck Configuration (dimensions, zones, stairs)
  Step 2: Materials & Options (decking type, railing, etc.)
  Step 3: Review & Download (PDF generation, compliance check)

GUIDE SYSTEM (steps.js):
- _guidePhaseMap defines phases with messages, tips, sections, actions
- Each phase shows/hides UI sections and presents action buttons
- Key phases for Step 0:
  address_lookup -> address_verifying -> footprint_loading -> verify_extracted
  -> site_elements_check -> ...
- "footprint_loading" phase (S73): prevents user interaction until building
  footprint API completes
- Overpass auto-retries once on failure, shows warning banner if both fail

AI CHAT (built into guide):
- User can type questions/instructions in the chat input
- Backend (main.py build_ai_helper_prompt) builds context-aware system prompt
- AI can return ACTIONS that modify the p state
- Action types: setParam, addZone, cutoutAdd, addStair, resetHousePosition, etc.

================================================================================
7. ENGINE AND STRUCTURAL CALCULATIONS
================================================================================

engine.js calculates all structural members per IRC 2021:
- Joist size from Table R507.5 (span vs spacing vs load)
- Beam size from Tables R507.5(1)-(4) (effective joist span)
- Post size (4x4 or 6x6) based on height
- Footing diameter and depth from frost zone
- Post count from beam span limits

calc_engine.py is the Python mirror for PDF output.

permit_checker.py runs 143 compliance checks against IRC R301-R507.
permit_spec.py is the single source of truth for all spec data rendered on PDF sheets.

Multi-zone calc (calcAllZones):
- Flush zones: joists only, no posts/beam (rim board acts as beam)
- Dropped zones: full structure (posts, footings, beam, joists)
- getEffectiveBeamType(zone, p): flush auto-converts to dropped when zone.h != mainH

================================================================================
8. ZONES AND MULTI-LEVEL DECKS (S60-S81)
================================================================================

ZONE SYSTEM:
- Zone 0 is the main deck (virtual, not in zones array)
- Users add zones (type: 'add') or cutouts (type: 'cutout')
- Zones attach to parent zones on specific edges
- Each zone has independent height, beam type, joists

KEY HELPERS (zoneUtils.js, all on window.*):
  getAllZones(p)           Main deck + all user zones
  getZoneRect(id, p)      Deck-local rect for any zone
  getCompositeOutline(p)  Boolean union for deck shape
  getExposedEdges(p)      Outer perimeter (for railing)
  getSharedEdges(p)       Internal edges between adjacent zones
  classifyHeightDelta()   IRC classification of height difference
  suggestRiserPlan()      Uniform risers + requirements

TRANSITIONAL STAIRS (S81d-e):
- _landsOnZoneId: null = grade, number = lands on that zone
- Rise computed from |fromZone.h - toZone.h|
- 3D renders stair bottom at target zone surface, not ground
- PDF labels "LANDS ON [zone]" instead of "CONCRETE LANDING"

================================================================================
9. PDF GENERATION PIPELINE
================================================================================

POST /api/calculate triggers PDF generation:
1. calculate_structure(params) runs structural calc
2. build_permit_spec(params, calc) builds spec dictionary
3. run_checks(params, calc, spec) runs 143 compliance checks
4. Each draw_* function renders one sheet using matplotlib
5. PdfPages combines all sheets into one PDF
6. PDF stored temporarily, download link returned

SHEETS (in order):
  A-0: Cover (3D render + project info + specs)
  A-1: Plan & Framing (deck plan + joist/beam layout)
  A-2: Elevations (4 views: front, back, left, right)
  A-3: General Notes (IRC code references)
  A-4: Structural Details (ledger, footing, stair, guard rail, post/beam)
  A-5: Site Plan (lot polygon, house, deck, setbacks, vicinity map)
  A-6: Deck Attachment Sheet (checklist)
  A-7: Materials List (optional)
